소아응급 AI Agent

⭐ 대표 · MS ICT 과정 · AI 파트 2026.06 – 2026.07

보호자의 말에서 위험 신호를 선별하고 안전한 상담과 병원 안내까지 연결하는 에이전트

왜 — 새벽에 아이가 아픈 보호자에게 주어진 건 검색창뿐입니다. 하지만 의료에서 LLM의 환각은 곧 사고입니다. 이 프로젝트의 진짜 과제는 "AI가 함부로 말하지 못하게 만드는 것"이었습니다.

부딪힌 문제와 해결

LLM이 진단하듯 답해버리는 위험

→ 모델보다 먼저 실행되는 45개 결정형 안전 Rule. 위험 신호는 코드가 즉시 판단하고 LLM은 그 다음에만 발화. 오탐·미탐 감사를 반복.

근거 없는 답변을 어떻게 막을까

→ 승인된 의료 문서만 담은 Azure PostgreSQL(pgvector) RAG로 출처를 강제.

"좋아진 것 같은데?"라는 감

→ 정확도·안전성 judge + adversarial 케이스로 모든 변경을 수치로 검증.

"숨을 쌕쌕거려요" — 구어체 보호자 표현

→ Kiwi 형태소 계층 + Qwen3.5-9B LoRA 파인튜닝(H100)으로 보호자의 언어를 이해.

기능을 더하는 개발보다 실패를 막는 설계가 어렵고 중요했습니다. 안전 도메인에서는 모델을 신뢰하지 않는 구조가 오히려 신뢰를 만듭니다.

ConnectBeat

캡스톤 디자인 · AI 파트 단독 2025.06 – 2025.11

커플 채팅에서 8가지 감정의 흐름을 읽는 감정 분석 서비스

왜 — 다툰 뒤에야 "그때 서운했어"라는 말이 나옵니다. 대화 속 감정을 데이터로 보여주면 서로를 이해하는 타이밍이 빨라질 수 있다고 생각했습니다.

부딪힌 문제와 해결

"응" 한 마디로는 감정을 알 수 없다

→ 직전 N개 발화를 [CTX] 토큰으로 잇고 화자 토큰 [GF]/[BF]를 설계해 대화 문맥째 학습·추론.

확률이 애매하게 퍼지고 사소한 말에 과민 반응

→ 키워드 부스팅·부정어 감쇠·샤프닝·중립 복귀 휴리스틱을 강도 조절 가능한 옵션으로 계층화.

모델은 있는데 앱은 어떻게 쓰지?

→ FastAPI 서빙 + AWS EC2 직접 배포, 백엔드와 1분 단위 타임라인 집계 API 규격 설계.

모델 정확도만큼 "팀이 쓰기 좋은 API"가 중요하다는 것 — 서빙까지가 AI 개발이라는 걸 여기서 배웠습니다.

증상 기반 병원·진료과 추천

솔데스크 파이널 · 백엔드/프론트 2025.10 – 2026.02

증상을 말하면 진료과와 가까운 병원까지 이어주는 의료 웹 서비스

왜 — 아플 때 정말 어려운 건 치료가 아니라 "어느 병원, 무슨 과로 가야 하는지"였습니다. AI만 하던 시야를 넓혀 백엔드와 프론트를 함께 맡았습니다.

부딪힌 문제와 해결

공공 병원 데이터의 형식 불일치와 누락

→ 국립중앙의료원 API 데이터를 정제해 병원 DB 구축. 데이터가 지저분하면 그 위의 모든 기능이 흔들린다는 걸 체감.

"가까운" 병원이 곧 "좋은" 병원은 아니다

→ 거리 계산 + 평점·리뷰 반영 추천 정렬, 예약·즐겨찾기까지 백엔드부터 화면까지 구현.

병원까지 가는 길 안내

→ 카카오·네이버 지도에 TMAP 도보 경로를 조합해 서비스별 한계 보완.

4인 팀 feature 브랜치·PR 협업(백엔드 37커밋). 데이터가 화면까지 닿는 전 과정을 처음 끝까지 책임져 봤습니다.
